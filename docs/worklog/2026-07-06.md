## 상세

### Kafka 이벤트 스트림 (커밋 `3e6e752`)
- 흐름: `/api/track/*` → `TrackingService` → **`EmailEventPublisher` 포트** → Kafka `mail.events`(campaignId 키) → worker `EmailEventProjectionListener`(@KafkaListener) → `email_events` 테이블(read model) → 기존 distinct 집계 그대로
- 원칙: **발송 상태·억제는 동기 유지, 참여 이벤트만 비동기** — 트래킹 엔드포인트가 더 이상 DB를 안 건드림
- 프로젝션은 append-only + at-least-once — 중복 배달은 distinct 집계가 흡수
- compose에 단일노드 KRaft 브로커(apache/kafka 3.8, host 앱은 localhost:9092)
- 검증: 오픈 2회+클릭 → Kafka 경유 → `opened=1, clicked=1` 정확 집계, 테스트 61개 그린(2개 클래스 publisher로 갱신)
- 발견: `email_events.url`이 psql에서 OID 숫자로 보임 — `@Lob` String이 Postgres에서 large object로 저장되는 문제(기능 무해, 배포 전 `@Column(length=...)` 전환 후보)

### 로그 관측 스택 이관 (mail-platform `0560e69` / logmonitor `2e912ec`)
- 원칙: **로그를 소비·분석하는 프로젝트가 수집·색인 인프라를 소유**한다. mail-platform은 `./logs/*.json.log` 파일 생산만
- logmonitor compose 기본 모드 = pg + OpenSearch 2.19 + Dashboards + opensearch-init(인덱스 템플릿) + Fluent Bit — 기존 "mail-platform의 OpenSearch를 공유"하던 어색한 의존 제거, standalone 프로파일의 중복 OpenSearch 2.10도 통합
- Fluent Bit tail 경로는 `MAIL_PLATFORM_LOGS` 환경변수로 재지정 가능(기본 `../mail-platform/logs`)
- 검증: 새 위치 기동 → 템플릿 자동 등록, 캠페인 발송 로그가 `serviceId=mail-worker`로 색인

### MailHog 수신거부 — 사실은 정상이었다
- suppressions 테이블에 이전에 클릭한 `alice@example.com (unsubscribe)`가 이미 있었음 = 그때 클릭이 성공했던 것
- 오해 지점 2개: ① 수신거부는 **바운스가 아니라 억제**(별개 경로·별개 사유) ② 기존 캠페인 지표는 소급 변경 안 됨 — **다음 캠페인**에서 `suppressed`로 잡힘
- 재현: 수신거부 → 재발송 → `suppressed=1` ✓
