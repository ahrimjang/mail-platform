### 이중발송 경합 수정 (커밋 `ed4847d`)
- 문제: 기존 멱등 가드(`!= PENDING → 스킵`)는 read-then-write라, 재배달이 원 소비자의 발송 도중 도착하면 둘 다 PENDING을 읽고 둘 다 발송하는 창이 있었음
- 해법: `claim(id, staleAfter)` — 단일 조건부 UPDATE(`WHERE id=? AND (PENDING OR 정체된 SENDING)`). DB가 행 단위 직렬화로 정확히 하나만 승리
- `FOR UPDATE SKIP LOCKED` 대신 조건부 UPDATE인 이유: 큐가 id를 콕 집어주는 구조(배치 스캔 아님) + 느린 SMTP 동안 DB 락 안 붙듦
- `SENDING` 상태 신설 → `completeIfDrained`는 `pending==0 && sending==0`으로 보정(조기 COMPLETED 방지)
- 정체 클레임(2분) 재클레임 허용 → 크래시 복구 유지. 잔여 한계: 2분 창 내 크래시는 창 경과까지 재시도 지연
- **검증**: 같은 행에 동일 UPDATE 동시 2발 → `UPDATE 1` / `UPDATE 0` 직접 증명
- **함정 발견**: `ddl-auto: update`는 enum에 값을 추가해도 기존 CHECK 제약을 안 넓힘 → dev DB는 ALTER 필요했음(커밋 메시지에 기록). Flyway 도입 근거 +1
