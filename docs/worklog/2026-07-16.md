# 작업 로그 — 2026-07-16

SaaS(멀티테넌트) 전환 결정 → 구현(V16) → 기획 전체 검토 → 검토 1순위(수신자 동의·페이지네이션) 수정(V17).
이어서 검토 체크리스트의 🔴 전부 소화(안전장치 묶음 V18 + 미터링), 전이 이력 로그 실험 후 회수, SES/SNS 웹훅 코드 완성. 커밋 8건 푸시.

## 오늘 한 일 (요약)

### 1. SaaS 전환 결정 — 과금 모델이 결정을 뒤집었다
"회사별로 서비스하려면 SaaS형 전환이 필요한가?"에서 출발해 4가지 모델(공유형 SaaS /
전용 인스턴스×N / 셀프호스팅 라이선스 / SaaS+BYO)을 구조·인프라 비용 부담·청구 방식으로
비교했다. 결론: **공유형 SaaS + BYO 커넥터** — 비용이 크고 변동성 높은 발송(SMTP/SES)·
저장(S3)을 고객 소유 계정에 연결하면 그 비용이 **고객에게 직접 청구**되므로, 사용량
미터링·정산 시스템 없이 플랫폼 구독료만 받으면 된다. "회사별 인프라 비용을 어떻게 나눠
청구하나"라는 문제 자체가 소멸. ROADMAP의 "하지 않을 것: SaaS 전환" 입장을 번복하고
경위를 문서에 남김.

### 2. 멀티테넌트 구현 (V16, `434bfde`)
- **가입 = 회사 등록**: 워크스페이스 생성 + 첫 계정 ADMIN. 관리자/운영자 2역할.
- **격리**: 루트 엔티티 5개(campaigns/contacts/lists/templates/suppressions)에
  workspace_id — 자식 테이블은 부모 경유. 목록은 스코프 쿼리, **by-id는 남의 것이면
  404**(존재 자체를 숨김), 대시보드/분석 집계는 캠페인 조인. 기존 데이터는 "기본
  워크스페이스"로 백필.
- **테넌트별 구독 결정**: `suppressions`·`contacts`의 이메일 유니크가
  `(workspace_id, email)`로 — A사에서 수신거부해도 B사 발송은 무관.
- **공개 경로**(추적/수신거부): 토큰→메시지→캠페인→워크스페이스 역해석. 바운스 웹훅은
  메시지 correlation이 없으면 **억제를 버린다** — 전 테넌트 오염 방지.
- **`WorkspaceContext` 포트**: 시그니처 폭발을 피하려고 서비스가 요청 컨텍스트에서
  테넌트/역할을 해석(JWT는 이메일만 유지 → 역할 변경이 토큰 재발급 없이 반영).
  워커는 이 포트를 쓰지 않고 캠페인 행에서 역해석(호출 시 예외 나는 가드 빈).
- **관리 페이지**(`/settings`, ADMIN 전용): 워크스페이스 이름 · **BYO 프로바이더 선택**
  (SMTP: MailHog/SES/SendGrid/자체, 저장소: 로컬/S3/NCP — 선택만 저장, 연동은 로드맵) ·
  멤버 추가/역할 변경(마지막 관리자 강등 차단 409).
- 잡은 버그: `api()`가 403을 세션 만료로 오인(역할 도입 전 가정) → 운영자가 로그아웃되던 것.
- E2E: Acme 가입 → 레거시 28건 대비 0건 격리, 남의 캠페인 404, 운영자 403·내비 숨김,
  Acme 캠페인 워커 발송 COMPLETED(테넌트 억제 판정 경로 포함).

### 3. 기획 전체 검토 → 체크리스트
회사 A의 여정(가입→수신자→템플릿→캠페인→파이프라인→분석→과금)을 따라 기획을 검토,
[REVIEW-product.md](../REVIEW-product.md)로 문서화. 요지: 파이프라인·구독 모델·타겟팅은
견고, 비어 있는 곳은 ① 안전장치(테스트 발송/발송 전 확인/빌트인 copy-on-write/created_by)
② 계정 수명주기(초대/재설정/비활성화) ③ SaaS 운영 축(미터링→테넌트 rate limit→SES 연동).

### 4. 검토 1순위 수정 — 수신자 동의 + 페이지네이션 (V17, `ebb016c`)
- **동의 기록**: `contacts.consent_source/consented_at`, 경로별 자동 스탬프
  (MANUAL/CSV_IMPORT). 레거시는 null → 드로어에 "기록 없음"으로 정직 표기.
- **페이지드 수신자 테이블**: 기존 화면은 전량 로드 + 수신자당 3콜(13명에 41요청 실측).
  `GET /api/contacts/page`가 DB에서 필터(검색/리스트/구독/참여도)하고 **배치 3쿼리**
  (멤버십 IN·옵트아웃 IN·억제 IN)로 행을 사전 병합 — 페이지당 1콜. 참여도 조건은 비율이
  계산값이라 랭킹 매치를 메모리에서 페이징(문서화된 경로). 페이저 UI + 300ms 디바운스.

### 5. 안전장치 묶음 — 체크리스트 🔴 소화 (V18, `94759d6`)
- **테스트 발송**: `POST /api/campaigns/test-send` — 샘플 변수로 렌더 + "[테스트]" 제목 접두,
  파이프라인 완전 우회(캠페인/메시지/지표 무기록). 새 캠페인 폼에서 A/B 변형 골라 발송.
- **발송 전 확인 모달**: 등록이 요약(대상·세그먼트 예상 인원·시점·기간·A/B)을 확인한 뒤에만 큐 등록.
- **빌트인 템플릿 copy-on-write**: 빌트인 수정 시도는 409, `POST /api/templates/{id}/copy`로
  사본 생성 후 편집. 템플릿 카드 클릭 = 복사 후 편집 흐름.
- **created_by**(V18): 캠페인에 등록 계정 이메일 스탬프, 상세 캠페인 정보 카드에 표시.
- 도중에 발견한 것: mail-api에 `spring.mail`이 없어 로깅 폴백 MailSender가 잡혀 테스트 발송이
  조용히 로그만 남기던 문제 — api yml에 SMTP 설정 추가로 해결.

### 7. 전이 이력 로그 — 실험하고 회수 (`2b0b553`)
"대기열 등록→발송 취소 같은 전이 이력을 보고 싶다"에서 출발해 V19 `message_status_log`
(append-only 전이 기록 + 타임라인 UI)까지 구현했으나, 부작용을 정직하게 계산해 보니
**행 수 2배, 발송 핫패스에 INSERT +1(IDENTITY라 미배치), 집계 스캔 2배** — 지금 규모에선
비용>가치로 판단해 **전체 회수**(마이그레이션 파일 삭제 + DB 수동 정리). ROADMAP Tier 3에
"재도입 시 배치 INSERT + 보존 정책 필수" 조건과 함께 보류로 기록. 만들었다 버린 게 아니라
**측정→판단→회수의 근거를 남긴 것**이 이 항목의 산출물.

### 8. SES/SNS 바운스 웹훅 — 코드 완성 (`cc9a81d`)
[TODO-ses-sns.md](../TODO-ses-sns.md)의 "코드 먼저" 순서대로, AWS 없이 검증 가능한 부분 전부:
- `POST /api/webhooks/ses` — SNS 봉투 수신(text/plain이라 raw body 파싱).
- **아마존 서명검증**(v1 SHA1/v2 SHA256 RSA, 인증서 URL은 `https://*.amazonaws.com`만, 캐시),
  로컬 데모용 토글 `APP_WEBHOOK_SNS_VERIFY_SIGNATURE`(기본 true).
- **구독 자동 승인** — SubscribeURL fetch도 아마존 도메인만(위조 메시지 SSRF 차단).
- **SES 파서** — Permanent→HARD_BOUNCE, Transient→SOFT_BOUNCE, Complaint→COMPLAINT로
  정규화, `mail.headers`의 `X-Mail-Message-Id` echo로 correlation. **`BounceService` 무변경** —
  05 문서에서 예고한 "어댑터만 추가" 설계가 그대로 회수됨.
- 검증: 단위 13건(로컬 RSA 키페어로 서명/변조/비아마존 거부) + 로컬 E2E(샘플 바운스 봉투 →
  메시지 BOUNCED + 테넌트 억제) + 기본값 기동에서 위조 서명 401 확인.
- 남은 것은 AWS 콘솔 준비(도메인 검증·샌드박스 해제·SMTP 자격증명·SNS 구독)뿐 — 코드는 환경변수 주입으로 끝.
