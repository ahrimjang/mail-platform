# 작업 로그 — 2026-07-09

대용량 처리(ROADMAP-scale) Tier 1 착수 — 부하테스트로 병목을 실증하고, 캠페인 생성 팬아웃과 발송 처리량을 개선했다. **측정 → 근본원인 → 수정 → 재측정**의 순서를 지켰다.

## 오늘 한 일 (요약)
1. **부하테스트 하네스**(①) — k6 `loadtest/baseline.js` + `RESULTS.md`. 캠페인 1건 생성→완료까지 생성지연·드레인·E2E 측정.
2. **③ 팬아웃 병목 실증** — 생성이 수신자 수에 선형(5k=12.1초 → 100만 명 외삽 40분+). 동기 팬아웃이 원인.
3. **③ 비동기 팬아웃 구현** — 리스트 캠페인은 잡 1건만 발행하고 즉시 반환, 워커가 배치로 확장. `create()` **O(1)**.
4. **③ 재측정**(#22) — 리스트 1k/5k/100k 생성이 **~50ms 상수**(5k 기준 12.1초→~240×). 100k `EXPANDING` 라이브 캡처.
5. **② 처리량 개선** — 매-발송 7×COUNT를 EXISTS로, 복합 인덱스, 리스너 동시성 1→8.
6. **② 재측정**(#23) — 5k 드레인 단일 워커 **20.5→157.7 msg/s(~7.7×)**, 이중발송 0.
7. 케이스 스터디 문서 `loadtest/CASE-async-fanout.md` 작성, 커밋·푸시.

## 상세

### ① 부하테스트 하네스
- `loadtest/baseline.js`(k6): 로그인→캠페인 생성→COMPLETED 폴링, `campaign_create_ms`·`drain_msgs_per_sec`·`e2e_seconds` 산출.
- 결과는 `loadtest/RESULTS.md`에 날짜·조건과 함께 누적 기록.

### ③ 팬아웃 병목 → 비동기화
- **실증**(#17~21): 생성이 선형(≈2.5~3ms/명). "API는 리스트 크기와 무관하게 즉시 반환" 원칙이 `CampaignService.create()`의 동기 팬아웃(N행 INSERT + N발행)에서 새고 있었다.
- **수정**: 리스트 캠페인은 캠페인 1행 저장 + **팬아웃 잡 1건** 발행 후 반환. 워커 `CampaignFanoutListener` → `CampaignFanoutService.expand()`가 `contacts`를 keyset 페이지(1,000)로 스트리밍하며 배치 저장+발행.
- **동시성**: 새 상태 `EXPANDING`. 팬아웃 시작 시 `QUEUED→EXPANDING` 원자적 claim(중복 잡 스킵=멱등). dispatch의 `markSending`은 `QUEUED→SENDING`만, `completeIfDrained`는 `SENDING`에서만 완료 → EXPANDING 중 조기완료 차단. 확장 완료 시 `EXPANDING→SENDING`.
- **애드혹 유지**: `recipients[]` 배열은 요청 크기로 상한이라 인라인 유지.
- **스키마**: `V6` — campaigns 상태 CHECK 제약에 `EXPANDING` 추가(V1/V4 선례).
- **재측정**(#22): 리스트 1k=44/56ms · 5k=61/39ms · 100k=66/64ms — 수신자 수와 무관한 상수. 측정 중 100k가 `EXPANDING`(2.9만 행까지 배치 스트리밍)으로 라이브 캡처, 조기완료 방지 확인.

### ② 발송 처리량
- **근본원인**: `completeIfDrained`가 발송마다 `countByCampaign`(상태별 7개 COUNT)을 돌림 — 캠페인이 클수록 O(N²). 리스너 동시성 기본 1.
- **수정**: (a) 드레인 체크를 단축평가 `hasPendingOrSending`(EXISTS 1개)로, (b) 복합 인덱스 `idx_msg_campaign_status(campaign_id, status)`(`V7`), (c) 리스너 `concurrency 1→8`(활성 소비자 12).
- **재측정**(#23): 5,000 드레인 단일 워커 **20.5→157.7 msg/s(~7.7×)**, E2E 244→31.7초. 워커 2대였던 #19(32)보다도 ~4.9× 빠름. **이중발송 0**(MailHog 정확히 5,000) — concurrency 12에서도 원자적 claim 유지.
- 정직한 기여 분해: 5k에선 동시성이 지배적, EXISTS·인덱스는 대형 캠페인 O(N²) 제거가 본 효과.
