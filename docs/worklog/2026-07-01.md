## 상세

### 인증 (JWT)
- `SignupRequest`/`LoginRequest`/`AuthResponse`, `AuthService`(BCrypt), `SecurityConfig`(stateless) + `JwtAuthFilter`
- `/api/auth/**`, `/api/health` 공개, 그 외 Bearer 토큰 필요
- 라이브 검증: 토큰 없이 `/api/campaigns` → 403, 발급 후 정상

### MVP 재기획
- Notifuse(Go+Postgres, 연락처/리스트/템플릿/브로드캐스트/트랜잭셔널/멀티프로바이더/트래킹)를 우리 스택으로 축소
- 우선순위 결정: **발송 + 추적/분석이 MVP 코어**, 연락처/템플릿은 뒤로 → `docs/MVP-PLAN.md`
- 마일스톤: M1 실발송 · M2 추적/분석 · M3 템플릿/개인화 · M4 연락처/리스트

### M1 실발송
- `SmtpMailSender`(MailHog, `mail.sender.type`로 Logging과 분기), HTML 본문
- `Suppression`(전역 do-not-send), 메시지별 `unsubToken` + `/api/unsubscribe/{token}`
- 발송 실패 → `BOUNCED` + 자동 억제. 상태: PENDING→SENT|FAILED|BOUNCED|SUPPRESSED
- 라이브 검증: 발송·수신거부·억제 스킵 전 구간 확인 (MailHog)

### M2 추적/분석
- 오픈 픽셀 `/api/track/open/{token}`(1x1 GIF), 클릭 `/api/track/click/{token}?u=`(302)
- `EmailEvent`(OPEN/CLICK) 스트림, 참여지표는 status 아닌 **이벤트 distinct 집계**
- `CampaignView`에 bounced/suppressed/opened/clicked 추가
- 라이브 검증: 오픈 2회→opened 1(유니크), 클릭→clicked 1

### RabbitMQ 발송큐 마이그레이션
- H2 폴링 제거 → `mail.exchange`→`mail.send.queue`(durable)→DLQ `mail.send.dlq`
- 생산자 `RabbitMailQueue`(mail-api), 소비자 `MailSendListener`(@RabbitListener)
- H2는 **상태 저장소**로 유지, 큐엔 `SendJob(messageId)`만. 소비자 멱등(`dispatchOne`이 PENDING만 처리)
- `MailDispatchScheduler`/`findPending` 삭제
- 라이브 검증: 관리 UI에서 publish 35 = deliver 35 = ack 35, 큐 잔량 0, DLQ 0

### 실제 바운스 처리 (웹훅)
- 설계: `docs/bounce-webhook-design.md` (헥사고날, 이메일 억제 + 메시지 correlation)
- 결정: **일반 정규화 엔드포인트 먼저**(운영 시 SES 어댑터 추가), correlation 포함
- `POST /api/webhooks/generic`(`X-Webhook-Token` 검증) → `BounceService`
- HARD_BOUNCE/COMPLAINT → 억제. `X-Mail-Message-Id` 헤더 주입 + 웹훅 messageId로 해당 메시지 BOUNCED + `EmailEvent(BOUNCE)`
- 라이브 검증: 잘못된 토큰 401 / 정상 202 / 메시지 SENT→BOUNCED / 재발송 시 억제 SKIP

## 아키텍처 결정
- **발송큐 = RabbitMQ** (작업 큐: ack/재시도/DLQ/throttle). Kafka는 "대량"이어도 발송엔 부적합(병목은 프로바이더 rate limit)
- **이벤트 스트림 = Kafka** (오픈/클릭/바운스) — 나중에
- 헥사고날 유지: 큐·발송·바운스 파싱은 어댑터, 코어는 포트/정규화 이벤트만
