# 결제 정책 (안) — 발송량 기준 플랜

> 2026-07-24 작성. 벤치마크: [스티비 요금제](https://stibee.com/pricing)(스타터 기준).
> 스티비는 **구독자 수** 기준 과금이지만, 우리 플랫폼은 V20에서 **월 발송량** 기준으로
> 결정했으므로([ROADMAP 설계 노트](ROADMAP-scale.md)) — 스티비의 **플랜 구조와 운영
> 정책(무료 시작, 즉시 상향/차기 하향, 결제 수단)** 을 이식하되 과금 축은 발송량으로 번역한다.
> 구현 체크리스트는 [REVIEW-product.md](REVIEW-product.md) 7절과 연동.

## 1. 원칙

1. **무료로 시작한다** — 가입 즉시 스타터. 카드 등록 없이 발송까지 되는 첫 경험
   (V20 전환의 근거였던 "가입하고 바로 보낸다"의 완성).
2. **청구 기준은 월 발송 성공 수(SENT) 하나** — 이미 구현된 미터링(`monthlySent`,
   매월 1일 리셋)이 곧 청구 수치. 구독자(연락처) 수·멤버 수는 청구 기준이 아니라
   **플랜별 한도**로만 쓴다(스티비와 반대 구도).
3. **한도는 파이프라인을 멈추지 않는 지점에서 집행한다** — 발송 중 컷오프 금지(아래 4절).

## 2. 플랜 (안)

| | **스타터** | **스탠다드** | **프로** | **엔터프라이즈** |
|---|---|---|---|---|
| 월 요금 | **무료** | 9,900원 | 29,000원 | 문의 |
| **월 발송량** (청구 축) | **1,000통** | 10,000통 | 50,000통 | 협의(전용 한도) |
| 연락처 수 | 500명 | 5,000명 | 50,000명 | 무제한 |
| 멤버 수 | 1명 (관리자) | 3명 | 10명 | 무제한 |
| 발송 속도 상한¹ | 5통/초 | 20통/초 | 50통/초 | 협의 |
| A/B 테스트 | — | 제목만 | 전체(승자 자동발송) | 전체 |
| 참여도 세그먼트 | — | ✓ | ✓ | ✓ |
| 예약 발송 | ✓ | ✓ | ✓ | ✓ |
| 발송 로그·분석 | 기본 지표 | + 링크 랭킹 | + 히트맵·퍼널 | 전체 |

¹ 발송 속도 상한 = 이미 구현된 테넌트 토큰버킷(V19)의 **플랜별 최대치**.
  현재는 관리자가 자유 설정하는데, 플랜 도입 시 "플랜 상한 이내에서 자기 설정"으로 바뀐다
  — 기존 기능이 그대로 플랜 게이팅 장치가 되는 지점.

숫자 근거: 스타터 1,000통 = 스티비 스타터의 실질 발송량(구독자 500 × 월 2회)과 등가.
유료 구간 가격은 스티비 최저 구간(8,900/29,000원)에 맞춰 시장 눈높이 유지.

## 3. 미터링과 청구

- **카운트 대상**: `SENT`만. FAILED/BOUNCED/SUPPRESSED/테스트 발송은 미과금
  (실패를 청구하지 않는다 — 신뢰 원칙).
- **주기**: 매월 1일 00:00(KST) 리셋. 조회는 현행 `monthlySent`(실시간 계산) 그대로.
- **청구 스냅샷**: 월 마감 시점에 발송량을 **스냅샷 테이블에 고정**하고 청구서는 그 수치로
  발행한다. 조회 시점 재계산 값으로 청구하면 사후 데이터 변동 시 분쟁이 되기 때문 —
  구현 선행 과제(REVIEW 7절 "청구서" 항목).

## 4. 한도 초과 정책 — 어디서, 어떻게 막나

스티비의 "초과 시 즉시 상향 결제" 흐름을 이식하되, 집행 지점을 우리 파이프라인에 맞춘다:

| 시점 | 동작 |
|---|---|
| 잔여 80% 소진 | 관리 페이지 배너 + 관리자 이메일 알림 |
| 100% 도달 | **신규 캠페인 등록 차단**(발송 전 확인 모달에서 잔여량과 함께 안내) + 상향 유도 |
| 진행 중 캠페인 | **끝까지 발송한다** — 도중 컷오프 금지 |

- 집행 지점은 **캠페인 등록 시**(애드혹은 create, 리스트는 create + 팬아웃 시 재검사 —
  예약 캠페인은 등록과 발송 시점이 다르므로). **디스패치 단위 검사는 하지 않는다**:
  10만 통 캠페인의 5만 번째에서 멈추면 반쪽 캠페인이라는 최악의 UX가 되고,
  핫패스에 쿼터 조회를 얹는 비용도 불필요.
- 초과 과금(통당 종량제)은 도입하지 않는다 — 예측 불가능한 청구서보다 "차단+상향"이
  단순하고, 스티비도 같은 선택.

## 5. 플랜 변경 (스티비 정책 이식)

- **상향**: 즉시 — 차액 결제 후 새 한도 즉시 적용 (한도 차단 해제의 경로).
- **하향**: 다음 정기 결제일에 적용. 단, 하향 대상 플랜의 한도(연락처 수 등)를 이미
  초과 사용 중이면 하향 불가 안내.
- **해지**: 다음 결제일까지 유지 후 스타터로 강등. 데이터는 보존(발송만 스타터 한도로).

## 6. 결제 수단·주기 (스티비 정책 이식)

- **월간**: 카드 정기결제만(빌링키 — PG는 토스페이먼츠 상정).
- **연간**: 카드 또는 계좌이체, 2개월분 할인(-17%). 계좌이체는 세금계산서 발행.
- **결제 실패(dunning)**: 3일 간격 3회 재시도 → 7일 유예(발송 유지+배너) → 스타터 강등.
  워크스페이스 상태 `active → past_due → downgraded` 도입 필요(REVIEW 7절).

## 7. 환불 (간소화)

- 미사용(해당 결제 주기 발송 0통) 시 7일 이내 전액 환불.
- 발송 이력이 있으면 일할 계산 없이 잔여 기간 유지로 갈음(월정액 특성상).
- 연간 결제는 미경과 개월 수 기준 환불(사용 개월은 월간 정가로 계산).

## 8. 구현 매핑 — 있는 것 / 만들 것

| 정책 요소 | 상태 |
|---|---|
| 청구 수치(월 발송량 미터링) | ✅ 구현됨 (`monthlySent`) |
| 발송 속도 상한 장치 | ✅ 구현됨 (토큰버킷 V19) — 플랜 상한 연동만 추가 |
| 사용량·요금 카드 | ✅ 있음 — 플랜명·잔여량·예상 요금 표시로 확장 |
| 플랜 저장(`workspaces.plan`) + 한도 검사 | ⬜ 신규 (마이그레이션 + 등록 시 검사) |
| 월 마감 스냅샷 + 청구서 | ⬜ 신규 (REVIEW 7절) |
| PG 연동·빌링키·dunning | ⬜ 신규 — POC에서는 **문서화까지만**, 구현은 범위 밖 |
| 기능 게이팅(A/B·세그먼트) | ⬜ 신규 — 플랜 필드 도입 후 서비스 앞단 검사 |

POC 데모 범위 권장: **플랜 저장 + 스타터 한도 집행 + 사용량 카드의 플랜/잔여량 표시**까지.
결제(PG)는 이 문서로 갈음한다.
