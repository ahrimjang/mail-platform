# 로드맵 — 대용량 처리 (처리량·확장성)

> 목표: 기능 개수를 늘리기보다 플랫폼의 본질인 **처리량·확장성**을 깊게 판다.
> 병목을 측정으로 찾고 → 고치고 → 다시 측정해 숫자로 확인한다. 목표 시점 8월.

## 원칙
- **측정이 먼저다.** "확장 가능하다"는 주장 대신 "워커 1대 X → 4대 ~4X msg/s" 같은 수치.
  확장성 작업에 들어가기 **전에** 측정 환경부터 만든다.
- **깊이 > 넓이.** CRUD 표면적을 넓히지 않고(한계효용 0), 확장성 한 축을 끝까지 판다.
- 기존 설계의 약점도 데이터로 드러나면 그대로 기록하고 고친다.

## 현재 갖춰진 확장성 기반
- 비동기 파이프라인: API 즉시 반환 → 워커가 큐 드레인
- RabbitMQ 경쟁 소비자 + **원자적 claim** (PENDING→SENDING 조건부 UPDATE) → 워커 N대에서도 이중발송 0
- Kafka 참여 이벤트 스트림, Flyway 스키마 관리, 토큰/집계 인덱스

## 작업 우선순위

### Tier 1 — 반드시 (완료 · [측정 기록](../loadtest/RESULTS.md) · [케이스](../loadtest/CASE-async-fanout.md))
- [x] **① 부하테스트 하네스 + 베이스라인** — k6 `loadtest/baseline.js` + `RESULTS.md`.
      create 지연·드레인 rate·E2E 측정 완료(#17~21). 여기서 ③·②의 병목이 데이터로 드러났다.
- [x] **② 워커 수평 확장 / 처리량** — 경쟁 소비자 2대에서 **이중발송 0**(원자적 claim), 다만 처리량 서브선형.
      진짜 레버는 per-worker 동시성 + 메시지당 DB 왕복 → 리스너 `concurrency 1→8` + `completeIfDrained`의
      7×COUNT를 EXISTS로 + 복합 인덱스(V7). 재측정: 단일 워커 **20.5→157.7 msg/s(~7.7×)**, 이중발송 0(#23).
- [x] **③ 캠페인 fan-out 병목 제거** — 리스트 캠페인을 **비동기 팬아웃**으로. `create()`가 팬아웃 잡 1건만
      발행하고 즉시 반환(`QUEUED`), 워커가 keyset 스트리밍 배치로 확장(`EXPANDING→SENDING`, V6). 재측정:
      create가 수신자 수와 무관한 **~50ms 상수**(#22). → "API가 리스트 크기와 무관하게 즉시 반환" 원칙 회복.

### Tier 2 — 강한 추가
- [x] **④ 발송 속도 제한(throttling)** — *2026-07-20 완료.* 워크스페이스별 토큰버킷
      (V19). 공유 상태는 Redis 대신 **Postgres 원자적 조건부 UPDATE**(claim 패턴 계보 —
      새 인프라 0, 병목 시 `SendRateLimiter` 포트 뒤 Redis 교체). 거절 메시지는 TTL 1초
      파킹 큐 경유 재진입. 실측 rate=3 → 정확히 3.0 msg/s, 무제한 테넌트는 동시 0.3초
      완료(noisy neighbor 해소). [logic/11](logic/11-send-throttling.md).
- [ ] **⑤ suppression 블룸필터 캐시** — 발송마다 억제목록 DB 조회 = 핫패스 병목.
      수백만 주소를 작은 메모리로 O(1) 판정(오탐 시 DB 재확인 → 오발송 없음).
- [x] **⑥ 메트릭 대시보드** — *2026-07-20 완료.* Micrometer(어댑터 계측만 — 코어 무관)
      → Prometheus → 파일 프로비저닝 Grafana 6패널(처리량/큐깊이/스로틀/SMTP p95/API/힙).
      스로틀 실측이 그래프로 재현됨(거부율≈재큐율). [logic/12](logic/12-metrics-grafana.md).

### Tier 3 — 여유 있으면
- [ ] `mail_messages`·`email_events` 테이블 성장 대비(파티셔닝, 인덱스·아카이빙 전략)
- [ ] ID 전략 전환(IDENTITY→시퀀스)으로 `saveAll` INSERT 배치 활성화 — 팬아웃의 페이지당 개별 INSERT 제거
- [ ] (보류) 메시지 상태 전이 이력 로그 — V19로 구현했다가 비용(행 2배·핫패스 +1 INSERT·집계 2배) 대비 회수해 되돌림.
      재도입 시 배치 INSERT 전제 + 보존 정책과 함께.
- [ ] 백프레셔/프리페치 튜닝(RabbitMQ prefetch, 소비자 동시성)
- [ ] 소프트바운스 재시도 정책(N회 누적 → 하드 취급)

## 하지 않을 것
- 템플릿 종류·연락처 필드 등 **CRUD 넓히기** — 본질(처리량)과 무관
- 프론트 치장 — 돌아가는 데모 + 아키텍처 다이어그램 + 측정 숫자가 우선

## 설계 노트 — SaaS(멀티테넌시) 전환 (2026-07-16 1·2단계 완료)

처음엔 "하지 않을 것"으로 분류했으나 BYO 커넥터 과금 모델(비용이 큰 발송·저장 인프라를
고객 계정에 연결해 비용이 고객에게 직접 청구되게 하고, 플랫폼은 구독료만 받는 구조)과
맞물려 전환을 결정했다. 아래 1·2단계가 V16으로 구현됐고(가입=워크스페이스+ADMIN, 역할
2단계, 루트 엔티티 스코핑, 공개 경로 토큰 역해석, `(workspace_id, email)` 유니크),
3·4단계가 남은 로드맵이다. 남은 항목의 체크리스트는 [REVIEW-product.md](REVIEW-product.md).

1. **데이터 격리 (1단계, 전환의 대부분)** — `workspaces` 테이블 + 루트 엔티티
   (`campaigns`/`contacts`/`contact_lists`/`templates`/`suppressions`)에 `workspace_id` FK.
   자식 테이블(mail_messages, email_events, memberships, list_unsubscribes)은 부모 경유로
   격리되므로 컬럼 불필요. JWT 클레임에 workspace를 싣고, 조회 쿼리에 스코프를 강제한다.
   헥사고날 구조 덕에 **포트 시그니처에 테넌트 컨텍스트를 추가하면 어댑터가 흡수**하는
   구조라 mail-core 유스케이스 로직 자체는 거의 그대로다.
2. **비인증 공개 경로의 테넌트 해석** — 추적 픽셀/클릭·수신거부·바운스 웹훅은 로그인이
   없다. 다행히 전부 **메시지 단위 토큰**으로 진입하므로 토큰→메시지→캠페인→workspace로
   역해석하면 된다(웹훅은 `X-Mail-Message-Id` correlation이 같은 역할). 억제/옵트아웃
   판정도 이 경로로 워크스페이스 스코프가 된다 — 전역 유니크였던 `suppressions.email`이
   `(workspace_id, email)` 유니크로 바뀌는 것이 핵심 변화.
3. **워크스페이스별 발송 신원** — From 도메인/SMTP(또는 SES) 자격증명, DKIM/SPF 검증
   플로우를 워크스페이스 설정으로. 현재의 `${ENV_VAR:기본값}` 전역 설정이 테넌트별
   레코드로 내려온다.
4. **운영 격리** — 테넌트별 발송 rate limit(Tier 2 ④의 토큰버킷을 워크스페이스 키로),
   플랜/쿼터(월 발송량), API 키 발급. 노이지 네이버 문제는 큐를 나누기보다 claim 시
   테넌트별 가중치로 시작하는 편이 변경이 작다.

구현 참고: 시그니처 폭발을 피하려고 JWT 클레임 대신 **`WorkspaceContext` 포트**(요청마다
DB에서 테넌트/역할 해석 — 역할 변경이 토큰 재발급 없이 반영)를 썼고, 워커는 이 컨텍스트를
쓰지 않고 캠페인 행에서 테넌트를 역해석한다.
