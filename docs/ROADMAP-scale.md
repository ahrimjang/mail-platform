# 로드맵 — 대용량 처리 (처리량·확장성)

> 목표: 기능 개수를 늘리기보다 플랫폼의 본질인 **처리량·확장성**을 깊게 판다.
> 병목을 측정으로 찾고 → 고치고 → 다시 측정해 숫자로 확인한다. 목표 시점 8월.

## 원칙
- **측정이 먼저다.** "확장 가능하다"는 주장 대신 "워커 1대 X → 4대 ~4X msg/s" 같은 수치.
  확장성 작업에 들어가기 **전에** 측정 환경부터 만든다.
- **깊이 > 넓이.** CRUD 표면적을 넓히지 않고(한계효용 0), 확장성 한 축을 끝까지 판다.
- 기존 설계의 약점도 데이터로 드러나면 그대로 기록하고 고친다.

## 현재 갖춰진 확장성 기반
- 비동기 파이프라인: API 즉시 반환 → 워커가 큐 드레인
- RabbitMQ 경쟁 소비자 + **원자적 claim** (PENDING→SENDING 조건부 UPDATE) → 워커 N대에서도 이중발송 0
- Kafka 참여 이벤트 스트림, Flyway 스키마 관리, 토큰/집계 인덱스

## 작업 우선순위

### Tier 1 — 반드시
- [ ] **① 부하테스트 하네스 + 베이스라인** — k6 또는 Gatling으로 50만~100만 수신자 캠페인 생성/발송.
      측정: API create 지연, enqueue 처리량, 워커 드레인 rate(msg/s), 엔드투엔드 시간, p99.
      → 모든 후속 작업의 기준선. **실제 병목 위치가 여기서 데이터로 드러난다.**
- [ ] **② 워커 수평 확장 검증** — mail-worker N개 인스턴스(경쟁 소비자). 처리량 선형 증가 + 이중발송 0 확인.
      claim 구조 덕에 코드 변경이 거의 없을 것으로 예상 — 예상대로인지 측정으로 확인.
- [ ] **③ 캠페인 fan-out 병목 제거** — 현재 `CampaignService.create()`가 리스트 전체를 메모리에 올려
      동기 처리(100만 수신자면 OOM/느린 응답). → 멤버 스트리밍/페이지네이션 + JDBC 배치 insert +
      fan-out 비동기화(캠페인 즉시 QUEUED 반환, 백그라운드에서 팬아웃).
      → "API가 리스트 크기와 무관하게 즉시 반환"이라는 설계 원칙이 실제로는 새고 있는 지점.

### Tier 2 — 강한 추가
- [ ] **④ 발송 속도 제한(throttling)** — 토큰버킷. 워커 간 공유 상태는 Redis.
      프로바이더 rate limit 대응에 필수인 분산 rate limiting.
- [ ] **⑤ suppression 블룸필터 캐시** — 발송마다 억제목록 DB 조회 = 핫패스 병목.
      수백만 주소를 작은 메모리로 O(1) 판정(오탐 시 DB 재확인 → 오발송 없음).
- [ ] **⑥ 메트릭 대시보드** — Micrometer→Prometheus→Grafana (처리량/큐깊이/지연 백분위).
      부하테스트와 짝을 이루는 관측 도구.

### Tier 3 — 여유 있으면
- [ ] `mail_messages`·`email_events` 테이블 성장 대비(파티셔닝, 인덱스·아카이빙 전략)
- [ ] 백프레셔/프리페치 튜닝(RabbitMQ prefetch, 소비자 동시성)
- [ ] 소프트바운스 재시도 정책(N회 누적 → 하드 취급)

## 하지 않을 것
- 템플릿 종류·연락처 필드 등 **CRUD 넓히기** — 본질(처리량)과 무관
- 프론트 치장 — 돌아가는 데모 + 아키텍처 다이어그램 + 측정 숫자가 우선
